# Product Requirements Document (PRD)

## Smart Hotel Experience Platform  
**Code Name:** *LuxStay Nexus*  
**Version:** 1.0

**Author:** UX-Engineering Team

## Executive Summary

LuxStay Nexus is a **single‑property, luxury hotel booking platform** that merges accommodation, in‑room dining, and ambient personalisation into one seamless, emotionally intelligent booking journey. The product leverages **Supabase** as its primary backend (database, authentication, storage, real‑time) and **Next.js (App Router)** for a high‑performance, SSR‑optimised frontend. Every pixel and interaction is engineered for a **“glassmorphism + premium minimalism”** aesthetic with obsessive attention to mobile‑first responsiveness and micro‑interactions that convey trust and elegance.

---

## 1. Product Vision & Goals

**Vision:**  
To make booking a hotel room feel less like a transaction and more like designing a private sanctuary. Guests control not only their room type and meals, but also the sensory atmosphere (temperature, lighting, music, mood) – all through a single, visually stunning interface that the hotel staff can see and prepare in real time.

**Business Goals:**

- Increase direct bookings (no OTA commissions) by 35%
- Boost ancillary revenue (food orders) by 25%
- Reduce check‑in friction and staff preparation time by 40%
- Achieve NPS > 70 through hyper‑personalisation

**User Goals:**

- Book a room in under 4 minutes on mobile
- Enjoy a cohesive, luxury brand feel across all touchpoints
- Personalise stay details once, and have them honoured automatically

## 4. Design System & UI/UX Strategy

### 4.1. Brand Identity – “Glassmorphism Elegance”

- **Color Palette:** Deep midnight blue (#0B1120) base, soft gold (#D4AF37) accents, pure white text, translucent overlays with `backdrop-blur-3xl bg-white/5`  
- **Typography:**  
  - Headings: Playfair Display (serif, elegant)  
  - Body/CTAs: Inter (clean, modern)  
- **Shapes:** Rounded-2xl cards, soft shadows (`shadow-2xl`), subtle border `border-white/10`  
- **Animations:**  
  - Page transitions: Framer Motion `AnimatePresence` with fade+slide  
  - Buttons: `scale` on hover/tap, smooth colour transitions  
  - Cards: 3D tilt on hover (using React‑Tilt)  
- **Illustrations:** Minimal line‑art icons, custom Lottie animations for loading states

### 4.2. Mobile‑First Approach (Non‑Negotiable)

- All screens designed at 375px breakpoint first, then scaled up  
- Booking form: stepper that collapses into a single‑screen carousel on mobile  
- Thumb‑friendly touch targets (min 48px), bottom‑sheet menus for extra options  
- Images lazy‑loaded with blurhash placeholders  
- Offline‑friendly: PWA ready with service worker for essential pages

### 4.3. Key UI Components (Design Specs)

- **Hero Section:** Full‑bleed video background, overlaid glass card with check‑in/out widgets. On mobile, video replaced by a high‑quality static image for performance.  
- **Room Cards:** Horizontal swipeable cards on mobile, grid on desktop. Each card reveals details on tap (expandable bottom sheet).  
- **Multi‑Step Booking Flow:**  
  - Step indicator (dots + labels) at top  
  - Each step is a full‑screen panel with progress feedback  
  - Smooth slide animation between steps  
- **Food Menu:** Horizontal category pills, grid of items with crisp images, quantity stepper, “Add to Order” with haptic feedback on mobile  
- **Preference Selector:** Mood‑based radio cards with large, expressive icons; sliders for temperature/lighting with live preview visualisation  
- **Checkout:** Summary card with room + food breakdown, glassmorphism receipt, Stripe Payment Element embedded seamlessly

## 6. Database Schema (Supabase PostgreSQL)

*All tables in `public` schema, Row Level Security (RLS) enabled where noted.*

### 6.1. Tables & Relationships

```sql
-- Enable UUID extension
CREATE EXTENSION IF NOT EXISTS "uuid-ossp";

-- Users (extends Supabase auth.users)
CREATE TABLE profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  full_name TEXT,
  phone TEXT,
  avatar_url TEXT,
  preferences JSONB DEFAULT '{}',
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Rooms
CREATE TABLE rooms (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  slug TEXT UNIQUE NOT NULL,
  name TEXT NOT NULL,
  category TEXT CHECK (category IN ('standard', 'deluxe', 'suite')) NOT NULL,
  description TEXT,
  base_price DECIMAL(10,2) NOT NULL, -- per night
  capacity INT DEFAULT 2,
  amenities JSONB, -- ["WiFi", "Mini Bar", ...]
  images TEXT[] DEFAULT ARRAY[]::TEXT[], -- array of Supabase Storage paths
  is_active BOOLEAN DEFAULT true,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Bookings
CREATE TABLE bookings (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  user_id UUID REFERENCES auth.users(id) NOT NULL,
  room_id UUID REFERENCES rooms(id) NOT NULL,
  check_in DATE NOT NULL,
  check_out DATE NOT NULL,
  guests INT DEFAULT 1,
  status TEXT CHECK (status IN ('confirmed', 'cancelled', 'completed')) DEFAULT 'confirmed',
  total_amount DECIMAL(10,2),
  payment_intent_id TEXT, -- Stripe PaymentIntent
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Food Orders (linked to a booking)
CREATE TABLE food_orders (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  booking_id UUID REFERENCES bookings(id) ON DELETE CASCADE NOT NULL,
  delivery_time TEXT CHECK (delivery_time IN ('at_checkin', 'scheduled')),
  scheduled_time TIME, -- only if scheduled
  status TEXT CHECK (status IN ('preparing', 'delivered')) DEFAULT 'preparing',
  notes TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- Food Order Items (line items)
CREATE TABLE food_order_items (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  food_order_id UUID REFERENCES food_orders(id) ON DELETE CASCADE NOT NULL,
  menu_item_id UUID REFERENCES menu_items(id) NOT NULL,
  quantity INT DEFAULT 1,
  price DECIMAL(10,2) -- snapshot of price at time of order
);

-- Menu Items (global hotel menu)
CREATE TABLE menu_items (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  name TEXT NOT NULL,
  category TEXT CHECK (category IN ('meal', 'snack', 'drink')) NOT NULL,
  description TEXT,
  price DECIMAL(10,2) NOT NULL,
  image TEXT, -- Supabase storage path
  is_available BOOLEAN DEFAULT true
);

-- Stay Preferences (1:1 with booking)
CREATE TABLE stay_preferences (
  id UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  booking_id UUID REFERENCES bookings(id) ON DELETE CASCADE UNIQUE NOT NULL,
  mood TEXT CHECK (mood IN ('romantic', 'work', 'relax')),
  ac_temperature INT CHECK (ac_temperature BETWEEN 16 AND 30),
  lighting TEXT CHECK (lighting IN ('dim', 'bright', 'warm', 'cool')),
  music_genre TEXT CHECK (music_genre IN ('jazz', 'classical', 'ambient', 'pop', 'none')),
  occasion TEXT,  -- e.g., 'birthday', 'anniversary'
  special_requests TEXT
);

-- Wishlist (BONUS)
CREATE TABLE wishlists (
  user_id UUID REFERENCES auth.users(id) ON DELETE CASCADE,
  room_id UUID REFERENCES rooms(id) ON DELETE CASCADE,
  added_at TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (user_id, room_id)
);
```

**RLS Policies (Examples):**

- Profiles: users can read/update their own row.  
- Bookings: users can see own bookings; admin (role) can see all.  
- Food orders: linked via booking, inherits booking visibility.

## 7. API Design (Next.js Route Handlers + Server Actions)

**Authentication:** All booking routes require valid Supabase session (checked via `createRouteHandlerClient`).

### 7.1. Endpoints

| Method | Route | Description | Auth |
|--------|-------|-------------|------|
| GET | `/api/rooms` | List rooms, optional date range for availability | Public |
| GET | `/api/rooms/[slug]` | Room details + availability for given dates | Public |
| POST | `/api/bookings` | Create booking (room, dates, guests) → returns booking ID | User |
| POST | `/api/bookings/[id]/food` | Attach food order to booking | User |
| POST | `/api/bookings/[id]/preferences` | Save stay preferences | User |
| POST | `/api/create-payment-intent` | Stripe PaymentIntent creation | User |
| POST | `/api/webhooks/stripe` | Stripe webhook (payment success/failure) | Public (signature) |
| GET | `/api/admin/bookings` | All bookings with status (admin only) | Admin |
| PATCH | `/api/admin/bookings/[id]` | Update booking/food/preferences status | Admin |
| GET | `/api/admin/analytics` | Revenue, occupancy stats | Admin |

**Server Actions (for forms):**

- `bookRoom()` – server action that checks availability atomically and inserts booking.  
- `addFoodToBooking()` – upserts food order items.  
- `savePreferences()` – inserts/updates stay_preferences.

All server actions use Supabase `service_role` key for RLS bypass where needed, ensuring safe server‑side operations.

## 8. Key Features & Implementation Details

### 8.1. Real‑time Room Availability

- On room selection, a server action queries `bookings` for overlapping dates: `(check_in < requested_checkout AND check_out > requested_checkin)` and status ‘confirmed’. Rooms with zero conflicts are available.  
- Results cached briefly with Supabase real‑time subscription to invalidate when a new booking is made.

### 8.2. Food Ordering with Combined Billing

- Food order items are stored with their price at the time of ordering (snapshot).  
- At checkout, a Stripe PaymentIntent is created for `room_total + food_total`.  
- Admin can mark food as “delivered” via dashboard; guest receives email notification.

### 8.3. Smart Stay Personalisation Engine

- **Mood Presets:**  
  - Romantic → dim warm light, 24°C AC, jazz music  
  - Work → bright cool light, 22°C AC, ambient instrumental  
  - Relax → warm dim light, 25°C AC, soft pop  
- User can override individual sliders after selecting a mood.  
- Preferences are displayed on admin dashboard 24h before check‑in so staff can prepare the room.

### 8.4. Admin Dashboard

- Protected route `/admin`; requires `is_admin` claim in JWT (set via Supabase custom claim or metadata).  
- Tabs: **Bookings**, **Food Orders**, **Room Readiness**, **Analytics**.  
- Uses Supabase Realtime to show live updates (e.g., new food order appears instantly).  
- Booking detail view shows guest preferences, food items, and status timelines.

### 8.5. Notifications

- **Booking Confirmation:** Email with summary, booking ID, QR code for check‑in.  
- **Food Status Change:** Email (and optional SMS) when food status changes to “preparing” / “delivered”.  
- **Pre‑Arrival Reminder:** 24h before check‑in, reminder email with weather forecast and custom preferences.

## 10. Performance & Optimisation Strategy

- **SSR / ISR:** Room pages statically generated, revalidated every 60s (or on‑demand via webhook when inventory changes).  
- **Image Optimisation:** Next.js `<Image>` with `unoptimized` for Supabase Storage URLs? Better to use Supabase Image Transformation (if enabled) with `?width=...` parameters. Alternatively, use Next.js custom loader to fetch from Supabase and cache.  
- **Lazy Loading:** All images below fold use `loading="lazy"`, Framer Motion `LazyMotion` for code splitting of animations.  
- **Code Splitting:** Dynamic imports for admin dashboard and heavy libraries.  
- **Mobile‑first CSS:** Tailwind’s responsive utilities with `sm:`, `md:`, `lg:` prefixes ensuring no desktop‑only bloat.  
- **Caching:** Supabase `@supabase/ssr` for server‑side queries, client‑side caching with SWR for frequently fetched data.

## 11. Security Requirements

- **Authentication:** Supabase Auth with Google OAuth + email. JWT stored in secure httpOnly cookie (via Supabase SSR helpers).  
- **Row Level Security:** All tables have RLS policies ensuring users only access their own data; admins use a separate Supabase client with `service_role` for server‑side data fetching.  
- **Input Validation:** Zod schemas on all API endpoints and server actions.  
- **Rate Limiting:** Upstash Redis‑based rate limiter on booking/payment routes (Next.js middleware).  
- **Stripe Webhook Signature Verification:** Stripe SDK verifies `stripe-signature` header.  
- **API Protection:** CORS restricted to the hotel’s domain. All mutations need valid session + CSRF (Next.js built‑in).

## 13. Development & Deployment Plan

**Phase 1: Foundation (Week 1-2)**  
- Set up Next.js project with Tailwind, Framer Motion, Shadcn/ui  
- Configure Supabase project (tables, RLS, Storage bucket)  
- Implement auth flows (Google OAuth, protected routes)  
- Build layout components, design system tokens

**Phase 2: Core Booking (Week 3-4)**  
- Room listing & detail pages with real‑time availability  
- Multi‑step booking flow (dates, guests, room selection)  
- Payment integration (Stripe Checkout)  
- Booking confirmation & email (Resend)

**Phase 3: Food & Personalisation (Week 5-6)**  
- Food menu UI, cart, attach to booking  
- Stay preferences wizard  
- Admin dashboard – bookings, food orders, preferences  
- Real‑time updates with Supabase Realtime

**Phase 4: Polish & Launch (Week 7-8)**  
- Performance optimisation (lazy loading, caching)  
- Mobile‑first QA, accessibility audit  
- PWA setup, offline fallback  
- Deployment to Vercel (frontend), Supabase Cloud (backend)  
- DNS, environment variables, monitoring

**Deployment Guide:**

1. Push code to GitHub, connect Vercel, set environment variables (`NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `STRIPE_SECRET_KEY`, etc.).  
2. Run Supabase migrations via CLI or Dashboard SQL editor.  
3. Set up Supabase Auth redirect URLs (production domain).  
4. Configure Stripe webhook endpoint pointing to `/api/webhooks/stripe`.  
5. Enable Supabase Realtime for `bookings`, `food_orders` tables.  
6. Optional: Use Supabase Edge Functions for sending SMS (Twilio) after food status change.
